# 코딩테스트 풀이 채점 (코드 검토 + 검증)

**핵심 원칙: "채점 = 제공된 예제를 한 번 돌려 답이 맞다"가 아니다.** 예제는 필요조건일 뿐이다. 진짜 채점은 _독립적으로 만든 정답 레퍼런스_ 와 _적대적으로 설계한 입력_ 으로 코드가 입력 제약 전체에서 옳은지를 증명하는 것이다. 예제만 돌려보고 "통과"라고 보고하지 말 것. 이 방법론은 문제 유형(DP·그래프·구현·문자열·수학…)과 저지(백준·프로그래머스…)에 무관하게 동일하다.

## 0. 문제 파악 (검증 설계의 기준)
대상 풀이 파일과 함께 다음을 먼저 확정한다. 모르면 사용자에게 묻거나 문제 링크를 참조:
- **출처/입출력 스타일**: 백준류(stdin→stdout 전체 프로그램) vs 프로그래머스류(함수 시그니처 + return값). 검증 방식이 달라진다(아래 2번).
- **입력 제약(범위)**: 각 변수의 상·하한, 원소 개수, 값 크기. → 전수 가능 여부와 오버플로 판단의 근거.
- **시간/메모리 제한**: 복잡도 판정 기준.
- **저지 종류**: 정답이 유일한가, 아니면 **스페셜 저지/다중 정답**(가능한 답 여럿)인가, **실수 오차 허용**인가. → 동등 비교 vs 검증기(validator) 선택.

## 1. 코드 검토 (정적 분석 — 빌드 전)
코드를 읽고 아래 _공통 체크리스트_ + _유형별 렌즈_ 로 점검. 발견 시 보고하고, 명백한 버그면 수정 제안.

**공통 체크리스트 (모든 유형):**
- **경계 조건**: 루프 부등호(`<`/`<=`), 배열 인덱스 범위, off-by-one, 베이스 케이스, 빈 입력·단일 원소.
- **자료형/오버플로**: 최악 입력에서 결과·중간 누적값이 자료형 범위를 넘는가(합/곱/팩토리얼은 `long long` 의심).
- **복잡도**: 입력 상한 대입 시 시간·메모리 제한 통과하는가. (예: O(N²)이 N=10^5면 위험, O(2^N)은 N 작을 때만.)
- **멀티 테스트케이스 상태 리셋**: 전역·정적 변수, 누적 컨테이너가 케이스마다 초기화되는가(누수 시 둘째 케이스부터 오답).
- **출력 형식**: 줄바꿈/공백/대소문자, 실수 정밀도(자리수), 다중 정답 허용 여부, 마지막 개행.
- **제어흐름의 견고성**: 스코프가 잘못된 early `return`/`break`(형제 분기를 통째로 자르는 패턴), 순서 의존 버그.
- **집계 방향**(재귀/탐색): 개수·크기는 위로(↑) 모으고(return/전역 누적), 깊이·현재상태는 아래로(↓) 인자로. 뒤바뀌면 오답.
- **MISRA/임베디드 관점**(사용자 선호): 동적할당 지양·정적배열, 미사용 변수, 매직넘버.

**유형별 렌즈 (해당 유형이면 추가로):**
- **DP**: 점화식의 정의·차원, 베이스 케이스(`dp[0]` 의미), 방문 순서(채워지기 전 참조 금지), 메모이제이션 초기값과 미방문 구분.
- **그래프(BFS/DFS/최단경로)**: 방문 처리 시점(큐 삽입 시 vs 꺼낼 때), 방향/무향, 자기 자신·중복 간선, 연결성 가정, 가중치 음수.
- **그리디**: "단순 선택이 깨지는 반례"가 없는지(정렬 기준·교환 논증). 의심되면 작은 입력 브루트포스와 대조.
- **이분탐색/매개변수탐색**: 경계 수렴(`lo`/`hi`/`mid`), 정수 오버플로(`(lo+hi)/2`), 판정함수 단조성, 종료 조건.
- **구현/시뮬레이션**: 좌표·방향 배열, 경계 밖 이동, 동시 갱신 vs 순차 갱신, 시간/턴 진행 규칙의 정확한 해석.
- **문자열**: 길이 0/1, 인덱스, 유니코드/대소문자, 부분문자열 경계.
- **정렬/자료구조**: 비교함수 strict-weak-ordering, 동률 처리, 스택/큐/우선순위큐의 빈 상태 접근.
- **수학/정수론**: 0·음수·1 같은 특이값, 나눗셈 0, 모듈러 연산 일관성, 정밀도.

## 2. 검증 (동적 — 독립 레퍼런스 대조)
풀이를 컴파일/실행 가능 형태로 준비:
- 백준류: 문제 폴더에서 `g++ -static -O2 -o boj_<번호>_test.exe boj_<번호>.cpp` 후 stdin으로 입력 주입.
- 프로그래머스류: 풀이 함수를 호출하는 작은 드라이버(`main`)를 스크래치패드에 만들어 테스트케이스를 직접 호출·비교.

아래 전략을 **문제 성격에 맞게 취사선택**(전부 다 할 필요는 없지만, 가능한 강한 것을 택한다):

1. **예제 TC** — 제공 입출력 일치(최소 조건, 이것만으론 불충분).
2. **독립 레퍼런스 대조 (가장 강력, 우선)** — 풀이와 _다른 방법_ 으로 정답을 내는 레퍼런스를 만들어 비교:
   - 입력 도메인이 작으면 → **전수**(모든 입력)를 레퍼런스와 1:1 비교.
   - 크면 → **랜덤 스트레스 테스트**: 작은 랜덤 입력 수천 개를 _브루트포스_ 와 대조해 첫 불일치(반례)를 찾는다.
   - 레퍼런스는 Python 등으로 "느리지만 확실히 맞는" 방식(완전탐색·정의 그대로)으로 작성.
3. **경계값** — 최소 입력, 제약 상한(최대 입력), 0/1/빈 입력, 단일 원소, 전부 동일 값, 최대/최소 절댓값.
4. **최악 부하** — 최대 크기 입력 + 최대 개수 테스트케이스로 (a) **시간 측정**(제한 대비) (b) **메모리** (c) 멀티TC면 **상태 리셋 누수**(모두 동일 최댓값 입력 → 모든 출력이 기대값인지).
5. **저지 방식별 비교 규칙**:
   - 정답 유일 → 문자열/수치 동등 비교.
   - **스페셜 저지/다중 정답** → 동등 비교 금지. 출력이 _제약을 만족하는지_ 검사하는 **검증기**를 작성(예: 경로면 실제 연결·길이 확인, 순열이면 조건 충족 확인).
   - **실수 오차 허용** → 절대/상대 오차 `eps` 내 비교.
6. **반례 주입** — 그리디/휴리스틱·까다로운 분기는 "깨질 법한" 입력을 의도적으로 설계해 주입.

랜덤 스트레스 테스트 골격(언어/문제 무관한 패턴):
```bash
# gen.py: 작은 랜덤 입력 생성 / brute.py: 느리지만 확실한 정답
for i in $(seq 1 2000); do
  python gen.py $i > _in.txt
  ./sol_test.exe < _in.txt > _a.txt
  python brute.py < _in.txt > _b.txt
  if ! diff -q _a.txt _b.txt >/dev/null; then echo "반례 발견:"; cat _in.txt; break; fi
done
```
> 시드를 `$i`로 고정해 재현 가능하게. 첫 불일치 입력이 곧 디버깅용 최소 반례.

## 3. 보고
**표 형태**로 검증 항목·방법·결과(✅/❌). 최소 포함:
- 레퍼런스 정합성(전수 또는 스트레스 N건), 경계(최소·최대·코너), 멀티TC 상태 리셋, 최악 부하(시간·메모리), 오버플로, 복잡도, 출력 형식.
- **"이 코드가 옳기 위한 전제"** 명시 — 어떤 입력 제약 덕에 안전한지, 제약이 커지면 무엇이 깨지는지.
- 발견된 버그/위험은 우선순위와 수정안과 함께. **코너에서 실패하면 ❌로 명확히 보고**(원인·반례·수정 제시). "대충 맞는 것 같다"로 넘기지 말 것.

## 4. 마무리
- **산출물 정리(필수)**: 검증으로 생긴 `*.exe`, 임시 입력/출력(`_in.txt`·`_a.txt`·`_b.txt`·`_big.txt` 등), 드라이버/레퍼런스 임시 스크립트 삭제.
  `find problems -name "*.exe" -delete` + 임시파일 `rm -f`. (임시파일은 가급적 스크래치패드에 만들어 격리.)
- 통과로 확인되면 "기록할까?"로 [record-problem](../record-problem/SKILL.md) 연계 제안.
- 커밋/푸시는 사용자가 명시할 때만.

## 제약 / 환경
- 빌드: g++ (WinLibs), `-static` 필수. 파일명은 ASCII `boj_<번호>.cpp`(한글 파일명은 g++/gdb에서 깨짐).
- 레퍼런스/제너레이터는 Python 등 가벼운 언어 권장(맞기만 하면 느려도 됨).
- 임시 파일은 만들고 반드시 청소.
